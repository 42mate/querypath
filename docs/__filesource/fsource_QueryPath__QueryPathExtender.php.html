  15.  * To create a new custom Query Path extension, create a new class that
  16.  * extends this, add your own methods, and then register it. The extension
  17.  * will then be automatically available.
  18.  *
  19.  * How this class works:
  20.  * 
  21.  * The extension mechanism for QueryPath uses a Decorator pattern.
  22.  * Essentially, each extension creates a new wrapper around the main QueryPath
  23.  * implementation. This class provides the generic decorator implementation. All other
  24.  * extensions should extend this class.
  25.  *
  26.  * Since extensions are decorators, they can reliably trap and override the behavior
  27.  * of functions. So you gain the main advantages of inheritance. But You also gain two
  28.  * major additional advantages: You get what equates to 'multiple inheritance'. Many
  29.  * different extensions can wrap a single QueryPath, essentially combining many
  30.  * QueryPath-like objects.
  31.  *
  32.  * The second advantage is explicit orderability. You can determine based on order of
  33.  * import which extensions will be fired last (first imported) and which will be fired
  34.  * first (last imported). This gives you an additional degree of control.
  35.  * 
  36.  * Of course, all of the downsides to multiple inheritance are present in a decorator
  37.  * pattern: You can have undefined behaviors when multiple decorator classes override
  38.  * the same base method. You can also have difficult-to-debug issues when many
  39.  * different classes *could* be responsible for errant behavior.
  40.  */
